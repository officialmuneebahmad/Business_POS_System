1. Introduction & Core Scope

Business POS System is a high-performance, lightweight, offline-resilient Point of Sale (POS) system designed specifically for non-technical retail users. Built using pure vanilla web technologies (HTML, CSS, JS), it removes framework bloat to achieve near-instantaneous page load speeds, high UI responsiveness, and compatibility across low-spec hardware.

Traditional POS systems introduce high complexity with SQL/NoSQL servers, local hosting environments, and expensive subscriptions. This system leverages Google Sheets as its central cloud database database, allowing shop operators to manage products, adjust inventory settings, view customers, and analyze sales figures directly inside a simple spreadsheet.

Core Project Design Objectives:

  • No-Code Backend for Operators: System owners can view records, edit data errors, and audit transactions in Google Sheets without interacting with SQL managers or raw admin command terminals.
  • High Offline Resilience: The system maintains state inside browser localStorage. Operators can continue creating carts, decrementing inventory, and browsing customer history even during intermittent network drops.
  • Secured Workspace: Strict protection mechanisms guard the client interface from code extraction, inspector console modifications, and unauthorized client replication.
  • Clean PDF Generation: Automated physical receipt printing and A4 invoice conversions occur client-side via asynchronous DOM-to-PDF rendering pipelines.

2. System Architecture & Data Flow

The application follows a Decentralized Client-to-Sheet (DCS) model. Instead of directly communicating with a standard SQL backend, the Vanilla JS frontend interfaces with two external services: Firebase Web Client SDK for session protection, and Google Apps Script acting as a REST API middleware mapping data calls to Google Sheets cells.

Data Sync and Operation Lifecycles:

  1. Authentication Flow: Upon page load, the Firebase observer monitors the login state. The dashboard elements remain hidden in the DOM, showing a simple centered login container. Only successful Firebase credential match calls display the POS application wrapper.
  2. Sync Fetch (GET Pipeline): When the operator triggers cloud synchronization, the client generates a GET request to the Google Apps Script Web App URL. The client appends a cache-busting timestamp param (t=Date.now()) to guarantee it bypasses intermediate CDN/browser cache proxies. The API returns a unified JSON containing product definitions, current stock tallies, orders history, and customer directories, which are then written into the local state.
  3. Transactions (POST Pipeline): Actions such as "Add Product", "Place Order", and "Update Stock" trigger asynchronous fetch POST requests containing JSON action payloads sent to the Apps Script endpoint. To prevent CORS pre-flight handshake blocking on various networks, the requests execute in no-cors mode with text/plain headers, ensuring safe writing back to the Google Spreadsheet rows.

6. Security: Authentication & Developer Safeguards

As a client-hosted POS, protecting proprietary script parameters, customer database tables, and inventory costing details requires proactive code protection.

1. Session Guard (Firebase Identity Platform)

The interface blocks page access to unauthorized visitors. Firebase Authentication regulates authorization. The client implements a state observer callback to show or hide the core workspace: 

auth.onAuthStateChanged((user) => {
    if (user) {
        document.getElementById('login-screen').style.display = 'none';
        document.getElementById('app').style.display = 'block';
    } else {
        document.getElementById('login-screen').style.display = 'flex';
        document.getElementById('app').style.display = 'none';
    }
});

2. Console & Inspect Guard (protect.js)

To prevent unauthorized inspection, modification of pricing, and source code cloning, the system deploys an active listener and defense monitor in protect.js:

  • Context Block: Right-click menus (contextmenu events) are explicitly disabled across the DOM.
  • Keyboard Shortcuts Intercept: Key listeners intercept and block shortcuts: F12, Ctrl+Shift+I, Ctrl+Shift+J, Ctrl+U (View Source), Ctrl+S (Save Page), Ctrl+A (Select All), Ctrl+C (Copy), and Ctrl+Shift+C.
  • Debug Timing Defense Loop: A recurrent worker executing every 1,000ms uses time-delta thresholds around a native debugger; call: 
setInterval(function () {
    const before = new Date().getTime();
    debugger; // Pauses execution if browser DevTools are open
    const after = new Date().getTime();
    if (after - before > 100) {
        // If developer tools pause execution, render blackout warning
        document.body.innerHTML = `<div class="blackout-screen">Developer Tools Detected</div>`;
    }
}, 1000);

7. Installation, Configurations & Troubleshooting

Part A: Deployment of Google Sheets Middleware

  1. Create a blank Google Sheet in your Google Drive. Name the file: Business POS Cloud DB.
  2. Navigate to Extensions > Apps Script.
  3. Delete any default code inside the editor. Paste the full codebase from Section 5 of this manual.
  4. Click Save Project (floppy disk icon).
  5. Click the Deploy button in the top right and select New Deployment.
  6. Choose type: Web App. Configure exactly as follows:
    • Description: POS Production Release
    • Execute as: Me (your-email@gmail.com)
    • Who has access: Anyone (This is required to allow Webhook calls from the client).
  7. Click Deploy. Authorize permission requests from the Google prompt warning.
  8. Copy the generated Web App URL (ends in /exec).

Part B: Web App Connection and Firebase Hook

  1. Launch the POS Pro Application in your browser or server hosting address.
  2. Log in using your registered operator credentials.
  3. Navigate to the Settings tab (gear icon).
  4. Paste the copied Apps Script URL into the Google Script URL field.
  5. Adjust your exchange rates (e.g. USD to PKR standard rates: 278), Sender Name, Phone, Address, and Website details.
  6. Click Save Settings. Click the Sync Cloud button in the top header. The local database will automatically populate headers in the Google Sheet.

Troubleshooting Checklist:

Issue Identified Likely Root Cause Recommended Resolution Action
Sync Failed Alert Apps Script Web App was deployed with access set to "Only myself". Create a new deployment in Apps Script. Modify the access dropdown to "Anyone". Redeploy and update the URL in Settings.
Cart item low stock prompt The quantity added to the cart exceeds the value inside the Stock catalog list. Navigate to Stock tab. Increase the product inventory quantity using "Qty Add" form, then sync.
Blackout developer tools screen A browser inspection panel, console window, or system debugger is active. Close the Chrome DevTools panels (F12 or Ctrl+Shift+I). Refresh the web browser page.
Invoices not downloading Pop-up blocks, script loading failures, or network issues preventing html2pdf.js download. Ensure the internet connection is active to load the cdnjs script CDN on start. Add permission exceptions for files download in Chrome Settings.
